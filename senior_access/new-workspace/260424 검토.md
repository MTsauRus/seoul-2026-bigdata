## 1. 보행 속도 기준 불일치 — 최우선 해결 과제

통합 발표자료에서 가장 먼저 눈에 띄는 구조적 문제다.  
팀 전체가 "노인 보행속도"를 서로 다른 값으로 쓰고 있다.

| 팀원   | 일반인   | 일반 노인    | 보행보조장치 노인 | 비교 대상                     |
| ------ | -------- | ------------ | ----------------- | ----------------------------- |
| 김성령 | 1.28 m/s | (미사용)     | **0.88 m/s**      | 일반인                        |
| 심재현 | 1.28 m/s | 1.12 m/s     | **0.70 m/s**      | 일반인                        |
| 이정태 | 1.20 m/s | **0.78 m/s** | (미사용)          | 청년                          |
| 양석준 | —        | —            | —                 | (속도 미적용, 400m/800m 버퍼) |

**문제점:**

- 보행보조장치 노인 기준 속도가 0.70 / 0.88 m/s로 나뉘어 있다. 같은 사람을 다른 속도로 계산한 것이므로, 두 분석을 나란히 놓으면 내부 논리가 무너진다.
- 이정태는 비교군을 "청년"으로 설정했다. 프로젝트의 핵심 대비는 "일반 성인 vs 보행 약자 노인"인데, 청년이라는 기준은 결을 달리한다.
- 양석준은 보행속도를 전혀 적용하지 않고 400m(10분) / 800m(20분) 버퍼를 사용했다. 이는 나머지 세 분석이 "30분 도달권"으로 수렴하는 것과 타임 프레임 자체가 다르다.

**권장 대응:** 발표 전까지 팀 차원에서 속도 표를 하나 확정해야 한다. 김성령이 인용한 한음 외(2020) 논문 기준을 전체에 적용하거나, 심재현 기준(0.70 m/s)을 채택하거나 — 어느 쪽이든 한 가지로 통일하고, 통일된 기준을 발표 슬라이드 첫 페이지에 명시해야 한다.

## 2. 분석 방법론 불일치

| 팀원   | 거리 계산 방식                                  | 출발점 설정                    |
| ------ | ----------------------------------------------- | ------------------------------ |
| 김성령 | OSM 보행 네트워크 + Dijkstra (실제 도로망)      | 행정동 중심점 426개            |
| 심재현 | OSM 네트워크 기반 추정 (infra_dashboard_1 기준) | 행정동 중심점 + 5개 대표지점   |
| 이정태 | **유클리드 거리** (직선 반경)                   | 행정동 중심점                  |
| 양석준 | **유클리드 버퍼** (직선 반경)                   | 복지시설 위치 (구 중심점 근사) |

### 이정태 (`01_medical_access.py:151–160`)

```python
d = np.sqrt((xy[:, 0] - cx) ** 2 + (xy[:, 1] - cy) ** 2)
young_list.append(int((d <= DIST_YOUNG).sum()))
```

네트워크를 타지 않고 EPSG:5179 투영 좌표에서 직선 거리를 계산한다. 한강, 철도, 고속도로 같은 단절 요소를 완전히 무시한다. 서울처럼 한강이 남북을 가르는 도시에서 직선 거리 기반 분석은 접근 가능 시설 수를 과대 계산한다. 특히 강변 지역 행정동에서 오차가 크다. 평균 손실률 55.9%라는 수치 자체는 주목할 만하지만, 방법론의 신뢰도에 주석이 필요하다.

### 양석준 (`analysis.py:167–183`)

```python
# 복지시설: 구 centroid 사용 (실주소 지오코딩 대신 구 중심점 근사)
welfare_merged = welfare_df.merge(
    gu_centroids[['구명', 'geometry']].rename(columns={'구명': '시군구명'}),
    on='시군구명', how='left'
)
```

복지시설 수천 개의 위치를 각 구의 중심점 **1개**로 모두 몰아버린다. 경로당 수백 개가 모두 종로구 중심점에 있는 것으로 처리된다는 뜻이다. 결과적으로 400m/800m 버퍼 분석 자체가 실데이터를 반영하지 못한다. `visualize_dong.py`에서는 Nominatim으로 개별 주소 지오코딩을 시도하지만, 핵심 분석 파이프라인인 `analysis.py`는 이 결함이 그대로 남아 있다.

## 3. 팀원별 산출물 평가

### 3-1. 김성령

**잘한 점**

- 프로젝트 전체 뼈대를 담당했다. 서울 전역 OSM 보행 그래프(162,440 노드)를 직접 구축하고 Dijkstra로 도달권을 계산한 것은 이 프로젝트에서 가장 기술적으로 탄탄한 부분이다.
- 이미 7개 지도를 하나의 대시보드(`260421_dashboard/index.html`)로 통합해 놓았다. Part A(보행권) / Part B(기후) 탭 분리, 전체화면 기능, 반응형 그리드까지 완성된 상태다.
- "손실률 × 영향받는 노인 수"로 정책 우선순위를 정량화한 `07c_priority_map.html`은 발표의 결론 슬라이드로 바로 쓸 수 있는 수준이다.

**지적 사항**

- `06_b1_heat_shelter_260420.py:21`에 경로가 `/Users/mtsaurus/Projects/seoul-2026-bigdata`로 하드코딩되어 있다. 실제 프로젝트 경로 (`2026seoul-bigdata`)와 다르다. 다른 환경에서 실행하면 즉시 오류 난다.
- Part B(기후 도메인 — 폭염쉼터, 한파쉼터, 결빙)가 Part A(보행권 격차)와 주제적으로 이어지는지 발표에서 논리를 명시해야 한다. 두 Part가 현재 탭으로만 구분되어 있어 "이 분석들이 왜 함께 있는가"에 대한 서사가 없다. 기후 인프라 접근성도 결국 노인 보행속도로 재단한다는 논리를 전면에 내세워야 한다.
- 결빙 취약 지도(10_b3)에서 열선 OSM 매칭률이 22.6%에 불과하다. 이 수치를 시각화에 표시하지 않으면 발표장에서 신뢰도 질문을 받을 수 있다.

- 횡단보도 신호 시뮬레이터는 관객 참여를 유도하는 요소로 발표에서 쓸 수 있다.

- `infra_별자리지도.html`은 Dark 테마를, `infra_dashboard_1.html`은 밝은 베이지 계열 테마를 쓴다. 같은 사람의 산출물인데도 스타일이 다르다. 통합 대시보드에서 한 방향으로 선택해야 한다.
- 별자리 지도가 데이터 기반인지, 더미/샘플 데이터 기반인지가 HTML 소스에서 바로 확인이 안 된다. 발표 전에 실데이터 연결 여부를 확인할 것.
- 보행속도 0.70 m/s를 사용하는 근거 논문이 코드 어디에도 명시되지 않았다. 김성령의 한음 외(2020)와 다른 출처를 쓴다면 발표 Q&A에서 설명이 필요하다.

### 3-3. 이정태

**잘한 점**

- 의료시설(병의원 + 약국) 접근성이라는 도메인 선택은 "노인이 만성질환 관리를 동네에서 할 수 있는가"라는 질문으로 직결되어 주제 설득력이 높다.
- 병의원 / 약국 / 전체를 레이어별로 분리한 구성, 구별 비교 차트(03_gu_comparison), 분포 히스토그램(04_distribution)은 분석의 층위가 다양해 발표 자료에 쓸 슬라이드가 많다.
- 평균 손실률 55.9%라는 결론 수치가 김성령의 54.7%와 거의 일치한다. 서로 다른 도메인에서 나온 숫자가 수렴하면 발표의 논거 강도가 높아진다.

**지적 사항**

- 가장 큰 문제는 방법론이다. 유클리드 거리로 도달권을 계산하면 한강 이남 동에서 강 너머 병원이 반경에 들어온다. 이를 그대로 발표하면 기술적 신뢰도에 직격타를 받는다. 최소한 "유클리드 거리를 사용했으며 실제보다 접근 가능 시설 수가 과대 추정될 수 있다"는 한계 주석을 지도와 슬라이드에 병기해야 한다.
- 비교군이 "청년(1.20 m/s)"이다. 다른 분석들이 "일반 성인(1.28 m/s)"을 기준으로 하는데, 이 차이가 발표에서 "왜 이쪽만 다른가"라는 질문을 유발한다. 통일하거나 이유를 명시해야 한다.
- 행정동 이름이 `full_name = gu_name + dong_code[-3:]` 형태로 코드 번호가 그대로 출력된다(`01_medical_access.py:85`). 지도 툴팁에서 행정동 이름이 숫자 코드로 표시되어 가독성이 나쁘다. 코드-이름 매핑 CSV가 없다고 주석에 적혀 있는데, 발표 전에 해결해야 한다.

### 3-4. 양석준

**잘한 점**

- 복지시설(경로당, 노인복지관)과 녹지(공원)를 묶어서 "생활 복지 + 쾌적성"을 함께 본 것은 다른 팀원 도메인과 겹치지 않는다.
- `welfare_analysis_chart2.png`, `park_analysis_chart.png`는 정적 이미지이므로 PPT에 곧바로 붙여 쓸 수 있어 발표 슬라이드 제작에 편의를 제공한다.
- 복지시설 확충 우선순위 TOP 5(`top5_urgent_districts.csv`)는 정책 제언 슬라이드에 바로 쓸 수 있는 결론이다.

**지적 사항**

- 핵심 분석(`analysis.py`)에서 복지시설 위치를 구 중심점으로 근사한 것은 본 프로젝트의 가장 심각한 방법론 결함이다. 경로당 수천 개가 구 중심에 몰려 있다고 가정하면, 해당 구의 접근성 분석 결과 자체가 의미를 잃는다. `visualize_dong.py`의 Nominatim 지오코딩을 핵심 파이프라인에도 적용하거나, 최소한 결과에 명시적 한계 표시가 필요하다.
- 400m / 800m 버퍼는 "도보 10~20분"을 가정하지만, 노인 0.88 m/s 기준 10분은 528m, 20분은 1,056m다. 버퍼 반경 수치가 노인 보행속도와 맞지 않는다. 일반 성인 1.28 m/s 기준 10분(768m)과도 다르다. 어떤 속도를 가정한 것인지 명시되어 있지 않다.
- 다른 팀원들이 모두 **30분** 기준을 쓰는데, 여기는 10~20분 버퍼를 쓴다. 타임 프레임이 달라 나란히 놓았을 때 "이 분석만 왜 짧은 시간이냐"는 질문이 나올 수 있다.
- 시각화 파일에 PNG 정적 이미지가 많다. 다른 팀원들이 모두 인터랙티브 HTML 지도를 제출한 것과 균형이 맞지 않는다.

## 4. 시각화 일관성 점검

### 4-1. 테마 충돌

| 팀원                       | 테마                       | 주요 배경색 |

| 김성령 (260421_dashboard)  | Dark                       | `#0a0a14`   |

김성령과 심재현의 별자리 지도는 같은 Dark 계열로 시각적으로 자연스럽게 붙는다. 반면 이정태와 양석준의 산출물은 라이트 테마다. 하나의 발표자료에서 두 테마가 섞이면 산만해 보인다. 통합 대시보드에서 Dark 혹은 Light 중 하나로 통일하거나, 섹션을 나누어 테마 전환을 의도적으로 설계해야 한다.

### 4-2. 서울 시각화 단위 혼재

| 팀원   | 단위                                      |
| ------ | ----------------------------------------- |
| 김성령 | 행정동 426개 (코로플레스) + 등시선 폴리곤 |
| 심재현 | 행정동 중심점 + 25개 자치구 보행권 지수   |
| 이정태 | 행정동(집계구 dissolve)                   |
| 양석준 | 25개 자치구                               |

양석준의 분석 단위가 구(25개)인 반면 나머지는 동(426개) 단위다. 정밀도가 다른 지도를 나란히 놓으면 "같은 서울인데 왜 저쪽은 더 세밀하냐"는 인상을 줄 수 있다.

### 4-3. "노인"의 정의 혼재

발표자료에서 "노인"이 0.70, 0.78, 0.88, 1.12 m/s 네 가지로 분열되어 있다. 청중 입장에서는 "노인"이 누구인지 혼란스럽다. 발표 오프닝에서 속도 그룹을 그래픽으로 한 번 정리하고, 각 도메인이 어느 그룹을 기준으로 했는지 명시하는 슬라이드가 필수다.

1. **속도 통일 회의**: 팀 전체가 동의하는 속도 테이블 확정. 최소한 보행보조장치 노인 기준치 하나를 합의해야 한다.
2. **이정태 — 동명 매핑 수정**: `full_name`이 코드 번호로 출력되는 문제 수정 필요. 지도 툴팁이 불완전한 상태로 발표할 수 없다.
3. **양석준 — 복지시설 지오코딩 재검토**: 구 중심점 근사를 쓸 것이라면 분석 결과가 아닌 분포 현황 시각화로 역할을 제한하고, 도달권 분석은 Nominatim 지오코딩이 완료된 `visualize_dong.py` 산출물을 메인으로 삼는 것을 권장한다.

### 통합 대시보드 작업

4. **기준 대시보드 선택**: 김성령의 `260421_dashboard/index.html`이 가장 완성도가 높으므로 이를 기반으로 확장하는 방식이 효율적이다.
5. **도메인별 지도 추가**: 심재현·이정태·양석준의 핵심 지도를 김성령 대시보드에 새 탭 또는 새 섹션으로 추가한다.
6. **테마 통일 결정**: Dark 테마 유지 시 이정태/양석준 산출물을 재스타일링하거나, 섹션 전환을 명시적으로 설계한다.

### 발표자료 구성 제안

7. **서사 구조**: "속도 격차 정의(김성령 Part A) → 도메인별 피해 (의료·인프라·복지·기후) → 정책 우선순위(김성령 07c + 양석준 TOP 5)" 순서로 이어가면 논리 흐름이 자연스럽다.
8. **속도 표 슬라이드**: 발표 초반에 4개 도메인이 사용한 속도/방법론을 한 표로 보여주고, 차이가 있으면 그 이유를 명시한다.
