## 이전 작업(260420~260421) 대비 배경

260420 B1/B2/B3는 도메인별로 개별 HTML을 생성했다. 각각은 시각화로는 유효했지만, 세 도메인을 동시에 비교하기 어렵고 보행 속도 유형별 분석(일반인·노인·보행보조)이 없었다. 260428 작업의 목표는 이 세 도메인을 하나의 대시보드로 통합하고, OSM 보행 네트워크 기반 Dijkstra 계산으로 속도 유형별 쉼터 접근성을 정량화하는 것이다.

---

## 핵심 분석 방법론

### OSM 보행 네트워크 + Forward Dijkstra

서울 426개 행정동 centroid 각각에서 단방향 Dijkstra를 실행했다.

```
최대 탐색 거리: 1.28 m/s × 45분 × 60초 = 3,456 m
보행자 유형:
  일반인         1.28 m/s  (국제 보행자 속도 표준)
  일반 노인      1.12 m/s
  보행보조 노인  0.88 m/s
  보행보조 하위 15%  0.70 m/s
```

각 centroid에서 Dijkstra로 도달 가능한 OSM 노드 집합을 구하고, 더위쉼터(4,107개)·한파쉼터(1,642개)의 nearest node와 비교해 각 (보행자 유형, 시간) 조합별 도달 쉼터 수를 계산했다.

### 인구 데이터 매핑 문제 해결

행정동 shapefile(BND)과 인구 CSV(KOSIS)의 자치구 코드 체계가 다르다.

```
BND:   11010(종로) → KOSIS: 11110(종로)
BND:   11020(중구) → KOSIS: 11140(중구)
...
```

동 코드 suffix도 불일치해 단순 merge 시 426개 중 209개만 매칭됐다. 해결책: 구 단위 총인구에서 이미 매칭된 동의 인구를 빼고, 남은 인구를 미매칭 동 수로 균등 배분. 결과적으로 426개 동 전수 커버(65세 이상 합계 1,441,521명).

### Convex Hull 보행권 모식도

각 동·구의 Dijkstra 도달 노드 좌표(EPSG:5179 투영)를 Shapely `convex_hull`로 묶어 centroid 기준 상대 좌표(dx, dy)로 저장. HTML에서 현재 선택 지역의 hull을 Leaflet 지도 위에 직접 오버레이해 "어느 동의 보행권인지"를 지도와 연결해 보여준다.

## 개선 2 — 결빙 취약구역 폴리곤: 3단계 방법론 진화

### 260420 방식

서울 전체 경계에서 제설함 커버리지를 차감. 한강·산지가 모두 "결빙 취약"으로 포함됨.

### 260421 방식 (참고)

`clean_seoul = 행정동 union - OSM 수역·산림` → 제설함 버퍼 차감. 한강·산지 제외에 성공했으나, Shapely `difference` 계산 결과에 **삼각형 잔여물(artifact)** 이 다수 발생. 원 3개가 서로 접하는 지점의 삼각형 틈새가 작은 폴리곤으로 남는 문제.

### 260428 신규 방식 — Canvas destination-out

폴리곤 차집합 연산을 완전히 제거. JS 캔버스 레이어에서:

```
1. 서울 행정동 union 외곽선을 빨간색으로 전체 칠함
2. 제설함 10,437개의 원을 globalCompositeOperation = 'destination-out'으로 지움
3. 지우기는 멱등(idempotent) — 원이 겹쳐도 이중 도색 없음
```

결과: 삼각형 잔여물 완전 제거. 원이 하나라도 덮이는 영역은 모두 투명 처리. 파일 크기도 줄었다(4.6 MB → 3.7 MB).

## 개선 3 — 보행권 모식도: 독립 캔버스 → Leaflet 지도 오버레이

### 기존

오른쪽 패널에 별도 캔버스를 그려 보행권을 시각화. 실제 지도와 연결이 없어 어느 지점을 모식화한건지 불명확했다.

### 변경

Convex Hull을 Leaflet 지도 위에 직접 폴리곤으로 오버레이.

```
EPSG:5179 상대 좌표 (dx_m, dy_m)
→ dlat = dy / 111111
→ dlng = dx / (111111 × cos(lat_rad))
→ L.polygon([lat + dlat, lng + dlng], ...)
```

4개 보행 유형 hull을 반투명 레이어로 겹쳐 표시(현재 선택 속도 강조). 지도에 centroid 핀을 찍어 "이 핀이 모식도 기준점"임을 명시.

## 개선 4 — 창의 시각화 패널 정리

오른쪽 패널에 4가지 창의 시각화(🌌 중력궤도 / ✨ 보행소진 / 💓 생명선 / 🔗 단절망)를 개발했으나, 검토 후 실질적으로 정보를 잘 전달하는 두 가지만 남겼다.

| 유지        | 설명                                                                                  |
| ----------- | ------------------------------------------------------------------------------------- |
| 🌌 중력궤도 | 선택 지역 centroid 기준 동심원(속도별 도달 반경) + 쉼터 위치를 방위각·직선거리로 표시 |
| 🔗 단절망   | 중심→쉼터 광섬유 연결선: 도달 가능=빛나는 라인, 불가=단절 파편                        |

### 중력궤도·단절망 reach 판별 버그 수정

두 시각화 모두 초기 구현에서 **haversine 직선거리**로 "도달 가능 여부"를 판단했다. 마포구 상암동처럼 한강 인접 지역은 강 건너 쉼터가 직선거리상 가깝지만 보행 네트워크로는 실제로 갈 수 없다. 결과적으로 지도에는 "쉼터 0개 도달(빨간색)"로 표시되는데 단절망에는 "활성 광섬유 18개"가 그려지는 모순이 발생했다.

**수정 방식**: `getReach()`에서 OSM Dijkstra 실제 count(`cnt`)를 가져온 뒤, haversine으로 가장 가까운 `cnt`개 쉼터만 "도달 가능"으로 표시. 개수가 OSM 데이터와 일치하면서 위치 근사는 직선거리 기준(합리적 근사).

## 한계 및 유의사항

- **중력궤도·단절망 위치 근사**: OSM Dijkstra 도달 가능 쉼터의 개수는 정확하나, 시각화에서 "어느 쉼터"인지는 직선거리 순서로 근사. 실제 도달 가능한 특정 쉼터 목록은 캐시에 저장되지 않아 정확한 1:1 매핑 불가
- **결빙 취약 구역 산지 미제외**: Canvas 방식에서 서울 외곽을 행정동 union으로 사용 → 260421의 OSM 수역·산림 제외 로직 미반영. 산지·소규모 수역이 결빙 취약으로 표시될 수 있으나, 한강 본류는 행정동 경계에 미포함되어 자동 제외됨
- **구 단위 쉼터 도달 수**: 구 단위 집계 시 동별 평균으로 표시 (소수점 발생 가능). 소수 표시는 `Math.round()`로 처리
- **보행 네트워크 거리 = 경로 거리**: 실제 신호등 대기, 계단, 경사도 미반영
